## 관찰 / 문제

앱과 서버가 KIS 해외주식(미국) 전용으로 구현되어 있다.

- WebSocket TR ID가 `HDFSCNT0`(해외) 하드코딩
- 잔고 조회가 `CTRP6504R`(해외잔고) 전용
- `realtime_trading.market` 값이 NAS/NYS/AMS만 존재
- 국내 잔고·실시간 가격·주문 API가 하나도 없음

## 현재 구조 스냅샷

```
KIS WebSocket ──HDFSCNT0──▶ websocket_service.py ──▶ realtime.py (중계)
KIS REST ─────CTRP6504R──▶ kis_service.py ──▶ account.jsx (잔고)
realtime_trading.market: NAS | NYS | AMS
```

## 단계별 계획

---

### 0단계 — DB 스키마 변경

**파일**: `migrations/` 새 파일 추가

#### 변경 내용

1. **`realtime_trading.market` 허용값 확장**
   - 기존: `NAS`, `NYS`, `AMS`
   - 추가: `KRX`(코스피), `KOSDAQ`(코스닥)
   - CHECK 제약이 있다면 수정, 없으면 주석만 추가

2. **`realtime_trading` — `market_type` 컬럼 추가** (선택)
   - `market_type TEXT DEFAULT 'overseas'` — `'domestic' | 'overseas'`
   - market에서 자동 파생 가능하지만 명시하면 쿼리가 단순해짐

```sql
-- migrations/add_domestic_market_support.sql

-- market 컬럼에 국내 시장 값 허용
-- (CHECK 제약 없으면 이 ALTER는 생략)
ALTER TABLE realtime_trading
  DROP CONSTRAINT IF EXISTS realtime_trading_market_check;

ALTER TABLE realtime_trading
  ADD CONSTRAINT realtime_trading_market_check
  CHECK (market IN ('NAS','NYS','AMS','KRX','KOSDAQ'));

-- market_type 자동 계산 컬럼 (또는 뷰에서 처리해도 무방)
ALTER TABLE realtime_trading
  ADD COLUMN IF NOT EXISTS market_type TEXT
  GENERATED ALWAYS AS (
    CASE WHEN market IN ('KRX','KOSDAQ') THEN 'domestic' ELSE 'overseas' END
  ) STORED;
```

**체크포인트**: 기존 행 모두 `market_type = 'overseas'` 로 채워지는지 확인.

### 2단계 — 국내 실시간 체결가 WebSocket 연동

**목표**: realtime 탭에서 국내/미국 종목을 각각 구독·표시

#### 핵심: 하나의 WebSocket으로 국내+해외 동시 구독 가능

```
ws://ops.koreainvestment.com:21000  (동일 서버)
approval_key                         (동일 키)

국내 종목 구독: tr_id = "H0STCNT0", tr_key = "005930" (6자리)
해외 종목 구독: tr_id = "HDFSCNT0", tr_key = "AAPL"
```

#### 백엔드 `services/websocket_service.py`

```python
# 기존 subscribe_to_stock() — HDFSCNT0 하드코딩
# 변경 후: market 파라미터 추가

def _get_tr_id(market: str) -> str:
    return "H0STCNT0" if market in ("KRX", "KOSDAQ") else "HDFSCNT0"

async def subscribe_to_stock(self, ticker: str, market: str):
    tr_id = self._get_tr_id(market)
    # 구독 메시지 전송 (tr_id만 다름, 나머지 동일)

def _parse_domestic(self, data_str: str) -> dict:
    """H0STCNT0 응답 파싱"""
    fields = data_str.split("^")
    # 필드 순서: MKSC_SHRN_ISCD(0), STCK_CNTG_HOUR(1), STCK_PRPR(2),
    #            PRDY_VRSS_SIGN(3), PRDY_VRSS(4), PRDY_CTRT(5), ...
    return {
        "ticker": fields[0],          # 종목코드
        "price": float(fields[2]),    # 현재가
        "rate": float(fields[5]),     # 전일대비율
        "sign": fields[3],            # 1:상한 2:상승 3:보합 4:하한 5:하락
        "volume": float(fields[13]),  # 누적거래량
        "market": "domestic",
    }

def _handle_message(self, message: str):
    # 기존: HDFSCNT0 파싱
    # 추가: tr_id 확인 후 분기
    parts = message.split("|")
    tr_id = parts[1]
    if tr_id == "H0STCNT0":
        return self._parse_domestic(parts[3])
    elif tr_id == "HDFSCNT0":
        return self._parse_overseas(parts[3])  # 기존 로직
```

## 남은 의문

- 실전/모의 구분을 `is_mock` 플래그로 처리할지, DB 컬럼(`kis_credentials.is_mock`)에서 읽을지 — 현재 백엔드가 어떻게 구분하는지 확인 필요
- 국내 종목 코드 6자리(예: `005930`)와 해외 티커가 겹치지 않으므로 정규화 로직 충돌 없음 — 단, 앱 UI에서 국내 코드를 "삼성전자(005930)" 형태로 표시하려면 종목명 조회 API 별도 필요
- H0STCNT0 응답에서 필드 인덱스가 문서와 실제가 다를 수 있음 — 첫 연동 시 raw 메시지 로그 찍어서 인덱스 검증 필수
- 국내 잔고와 해외 잔고를 **하나의 통합 뷰**로 보여줄 것인지 vs **마켓별 분리** 유지할 것인지 — 현재는 분리 방향으로 계획
