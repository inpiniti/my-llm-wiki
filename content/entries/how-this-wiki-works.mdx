이 위키의 항목 하나는 [content/entries/](../../content/entries) 안의 `.mdx` 파일 하나에 대응한다.
파일을 추가하면 홈 화면 카드와 상세 페이지가 자동으로 생긴다. 빌드 단계나 DB 등록 같은 건 없다.

## 새 항목 만들기

`_TEMPLATE.mdx`를 복사해 `내-항목-이름.mdx`로 저장하면 끝이다.
파일명이 곧 URL slug가 된다 (`/entry/내-항목-이름`). `_`로 시작하는 파일은 목록에서 제외된다.

```bash
cp content/entries/_TEMPLATE.mdx content/entries/my-new-note.mdx
```

## frontmatter 필드

상단 `---` 블록이 메타데이터다. `title` `tldr` `status` `difficulty` `tags`는 카드와
필터에 쓰이므로 채우는 게 좋다. `project` `date` `impact` `decision`은 상세 헤더의
메타 그리드에 표시되고, `related`는 항목 간 링크를 만든다.

<Callout label="상태">
`status`는 세 가지다 — <b>solved</b>(해결됨), <b>wip</b>(진행 중), <b>open</b>(열린 문제).
홈 화면 필터와 통계가 이 값으로 동작한다. "아직 안 풀린 것"을 open으로 남겨두는 게 이 위키의 핵심이다.
</Callout>

## 본문에서 쓸 수 있는 것

표준 Markdown 전부 + 코드 펜스, 그리고 두 가지 커스텀 컴포넌트:

- `<Callout label="결론">…</Callout>` — 강조 상자
- `<Sidenote>…</Sidenote>` — 곁가지 메모

## agent와 함께 쓰기

핵심 동기는 이거다 — agent에게 작업을 시킬 때 이 위키를 먼저 참고하게 하는 것.
파일이 그냥 텍스트(MDX)라서 agent가 별도 도구 없이 바로 읽을 수 있다.
루트의 [CLAUDE.md](../../CLAUDE.md)에 "작업 전 위키 확인" 규칙을 적어두면, 같은 문제를
두 번 디버깅하지 않게 된다.

<Sidenote>
인터랙티브 데모(토크나이저·어텐션 시각화 등)는 디자인 샘플엔 있었지만 이번 골격엔 넣지 않았다.
필요해지면 본문에 클라이언트 컴포넌트로 끼워 넣을 수 있다.
</Sidenote>

## 남은 의문

frontmatter의 `related`를 양방향으로 자동 연결할까(지금은 단방향)? 태그 페이지가 필요할까?
검색은? — 항목이 쌓이기 시작하면 다시 판단한다.
