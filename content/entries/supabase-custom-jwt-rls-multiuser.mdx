## 관찰 / 문제

앱·백엔드가 "나 혼자" 전제로 설계돼, 비로그인으로 접속해도 남이 등록한 실시간 매매 종목이 전부 보였다. 원인은 `realtime_trading` 테이블의 RLS가 `Allow all access`(public/ALL/true) 정책이라 anon도 전체 조회가 됐던 것.

별도 회원 시스템은 없고, 사용자는 KIS(증권사) 계좌로만 로그인한다. 사용자 규모는 2~10명 소수.

## 왜 그런가 / 원인

Supabase Auth(이메일/소셜)를 안 쓰는데도 RLS의 `auth.uid()`로 사용자를 격리하려면, **Supabase가 신뢰하는 JWT를 직접 발급**해야 한다. Supabase는 프로젝트의 **Legacy JWT Secret**으로 HS256 서명된 토큰을 정식 인증으로 받아들인다.

격리를 켜는 과정에서 두 가지 함정이 있었다:

1. **무중단 순서** — RLS 정책을 먼저 켜면 기존 행(user_id NULL)이 전부 안 보이고 백엔드 감지도 막힌다.
2. **컬럼 타입 불일치** — `realtime_trading.user_id`는 `text`인데 `auth.uid()`는 `uuid`를 반환한다. `auth.uid() = user_id`(uuid=text)는 PostgreSQL에서 연산자 부재 에러.

## 해결

**① 결정적 user_id + 커스텀 JWT.** 계좌번호(CANO)로 `uuid5(고정namespace, CANO)`를 만들어 user_id로 쓴다(같은 계좌면 항상 같은 ID → 백필·추적 가능). KIS 토큰 발급 성공으로 본인을 검증한 뒤, 그 user_id를 `sub`로 하는 JWT를 표준 라이브러리만으로 HS256 서명한다.

```python
payload = {"sub": user_id, "role": "authenticated", "aud": "authenticated",
           "iat": now, "exp": now + ttl}
sig = hmac.new(SUPABASE_JWT_SECRET.encode(), signing_input, hashlib.sha256).digest()
# 앱은 이 토큰으로 Supabase 접속 → RLS의 auth.uid() == user_id
```

**② 무중단 전환 순서 (엄수).** RLS는 맨 마지막에 켠다.
1. 앱·백엔드를 user_id 기반으로 모두 배포 (이 동안 정책은 `Allow all` 유지 → 무중단)
2. 기존 행 백필: `UPDATE ... SET user_id = '<본인>' WHERE user_id IS NULL`
3. 그 다음에야 RLS 정책 적용
4. 백엔드는 RLS 켜면 anon으로 못 쓰므로 `SUPABASE_SERVICE_ROLE_KEY`로 우회 (HF Secrets 필수)

**③ 타입 캐스팅.** user_id가 text면 정책에서 `auth.uid()`를 캐스팅한다.

```sql
-- realtime_trading.user_id 가 text 인 경우
CREATE POLICY "rt_own_rows" ON realtime_trading
  FOR ALL TO authenticated
  USING (auth.uid()::text = user_id)
  WITH CHECK (auth.uid()::text = user_id);

-- 앱이 직접 안 쓰는 자격증명/키 테이블은 정책 없이 RLS만 → service_role만 통과
ALTER TABLE kis_credentials ENABLE ROW LEVEL SECURITY;
ALTER TABLE websocket_keys  ENABLE ROW LEVEL SECURITY;
```

<Callout label="결론">
Supabase Auth 없이도 Legacy JWT Secret으로 HS256 토큰을 직접 서명하면 RLS를 그대로 쓸 수 있다(`sub`=user_id, `role`/`aud`=authenticated). 외부 로그인을 인증원으로 삼을 때 강력한 패턴. 단 두 가지를 반드시 챙겨라 — (1) RLS는 "코드 배포 → 데이터 백필 → 정책 적용" 순으로 맨 마지막에 켜 무중단 전환하고, 백엔드는 service_role로 RLS를 우회한다. (2) `auth.uid()`는 uuid이므로 user_id 컬럼이 text면 `auth.uid()::text`로 캐스팅하지 않으면 정책이 깨진다. 테이블마다 user_id 타입이 다를 수 있으니 적용 전 `information_schema.columns`로 반드시 확인할 것.
</Callout>

## 남은 의문

테이블 간 user_id 타입이 text와 uuid로 갈린 건 앱(문자열 insert)과 백엔드(uuid 컬럼) 설계가 따로 자란 결과다. 장기적으로는 한쪽으로 통일(realtime_trading.user_id를 uuid로 ALTER)하는 게 깔끔하지만, 앱이 문자열로 insert하는 경로가 살아있어 캐스팅 방식을 택했다. 통일 시 앱 insert 경로 영향 검증이 선행돼야 한다.

<Sidenote>
같은 계좌인데 기존 종목의 user_id가 현재 로그인 user_id와 다른 사례가 있었다. uuid5는 결정적이라 같은 CANO면 같은 값이어야 하므로, 종목이 과거 다른 식별 시점/경로에서 등록됐다는 의미. 격리 전환 시 기존 데이터의 user_id 정합성을 한 번에 확인(GROUP BY user_id)하는 게 안전하다.
</Sidenote>
